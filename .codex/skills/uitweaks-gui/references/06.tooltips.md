# Tooltip Layer

Floating tooltip support, layered as a **separate renderer** so tooltip content can extend beyond its parent component bounds and beyond the dialog's own Cairo surface.

**Pieces:**
- `GuiTooltip : GuiNode` — sealed wrapper. **Layout-transparent**: implements only `IGuiNode` (not `IGuiComponent`), so the layout pass treats it as invisible (see `05.layout-parameters.md`). The wrapper has no layout parameters, occupies no layout space, and its `Content` children flow at the parent's declaration site as if they had been declared directly. `TooltipContent` is the floating fragment; `ConfigureBackground` tunes the auto-wrapping panel. The wrapper's bounds (used for hover hit-testing) are the union of its content children's allocated rectangles along the parent's flow axis.
- `GuiTooltipBackground : GuiContainer` — vanilla `GuiElementHoverText.DefaultBackground` recipe: `DialogStrongBgColor` fill, `DialogBorderColor` stroke at width 3, radius 1, 5px content padding.
- `TooltipHost` (public class, **cascading value**) — published at every `GuiDialog`'s root. Owns a per-arrange tooltip-region table that `GuiTooltip.Render` populates during the current combined arrange/paint walk. `TooltipHost.UpdateHover(lx, ly)` is called from `DialogInputDispatcher.DispatchMouseMove` (uncaptured branch) and forwards changes to the dialog's tooltip `FloatingLayerRenderer`.
- `FloatingLayerRenderer` (internal) — shared renderer used by both tooltip and overlay layers. Owns its own `GuiRenderTreeBuilder`, `ImageSurface`/`Context`/`LoadedTexture`, active fragment, and placement. For tooltips, `TooltipHost` calls `Show` with cursor-anchored placement at `cursor + (10, 15)` logical px, scaled, with screen clamping and bottom-overflow flipping for vanilla parity.

**Cascade wiring:** `DialogRenderer` ctor seeds `_builder.CascadeChain` with `(typeof(TooltipHost), null, host)`. The same chain is forwarded to the tooltip `FloatingLayerRenderer` via `SetCascadeChain` so tooltip content can also consume any dialog-root cascading value. The dialog component itself can't see the host (its handle has `parentBuilder = null`), but doesn't need to — only descendants do.

**Hover model:** Independent of the existing single-region `_hoveredToken` (which would otherwise be shadowed by an interactive child like a button). `TooltipHost.UpdateHover` walks its own region table in reverse and selects the topmost match. Tooltip is hidden on `MouseDown` (suppressed during interaction) and on dialog `Close`.

**Frame loop addition:** at start of the dialog's dirty combined arrange/paint walk, `_tooltipHost.ResetFrame()` clears the region table; `GuiTooltip.Render` repopulates via `host.AddRegion(this, bounds, TooltipContent, ConfigureBackground)`. After the main blit, the dialog renderer iterates `_floatingLayers` and calls `Render()` on each — for the tooltip layer this reconciles + redraws + blits the tooltip surface (uses the base `FloatingLayerRenderer` default lifecycle; `OnFrameStart` / `RunWalk` are no-ops).

**Surface lifecycle:** the tooltip surface is allocated lazily in the base `Update()` once a tooltip is active. Re-allocated only when the GUI scale or measured size changes (typical tooltips: < 320×60 logical px). `FloatingLayerRenderer.Schedule()` (inherited from the base) collapses scoped rebuilds into the dirty flag — the tree is shallow enough that rebuild-from-scratch is cheap.

**Usage:**

```csharp
builder.AddTooltip(0,
    tooltip: tb => tb.AddLabel(0, "Description goes here"),
    content: cb => cb.AddButton(0, "Hover me",
        onClick: DoStuff,
        widthMode: GuiSizeMode.Fill));
```

The wrapper carries no layout parameters — set them on the inner content components instead. The tooltip surface auto-sizes to its content. Pass `configureBackground:` to retune the wrapping panel without replacing it; for full chrome replacement, declare a custom panel inside the `tooltip` fragment (the auto-wrapper still adds 5px padding by default — set `configureBackground: bg => bg.LayoutParameters.Padding = new GuiThickness(0)` to defeat that).

**Constraints (current iteration):**
- Tooltip-internal interactivity is **not dispatched** because the tooltip layer has no `InputHost`. Tooltips are visual passthrough.
- Tooltips do not nest hover-update across multiple `TooltipHost`s — only one host per dialog. Nested tooltips inside tooltip content are not currently supported.
- Position follows the cursor; there is no anchor-to-trigger mode. Add later by storing the trigger bounds in the active region and using a custom `FloatingLayerPlacement.Anchor`.
