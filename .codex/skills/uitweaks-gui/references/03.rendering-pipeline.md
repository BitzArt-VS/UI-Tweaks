# Rendering Pipeline

`DialogRenderer` registered as `IRenderer` on `EnumRenderStage.Ortho`.

**Invalidation model:** the renderer tracks three cascading dirty states:

1. Reconcile — scheduled by `RequestReconcile()`. Runs pending render fragments through `ScopedRebuildQueue`, then requests arrange.
2. Arrange — scheduled by `RequestArrange()` or by reconcile. Recomputes layout/input regions through the current combined tree walk, then requests paint.
3. Paint — scheduled by `RequestPaint()` / `RequestRender()` or by arrange. Redraws the Cairo surface from the latest arranged bounds and uploads the texture.

Arrange and paint are split in the dialog renderer. `ExecuteArrangeWalk` clears and re-registers input regions, recomputes layout, caches each slot's arranged bounds, paints the Cairo surface, and uploads the texture. `ExecutePaintWalk` resets paint-time tooltip regions, repaints from the cached arranged bounds through `GuiRenderTreeBuilder.Paint`, and uploads the texture without recomputing layout or input regions. Unchanged frames only blit the previously uploaded texture.

**Cairo surface:** physical-pixel-sized `ImageSurface` + `Context` allocated at dialog construction (`logicalW * GUIScale` rounded). Recreated on the fly when `RuntimeEnv.GUIScale` or the dialog's logical size changes. The CTM is scaled by `GUIScale` during paint so component code can work in logical pixels. On dirty paint, the surface is cleared (transparent), components draw into it, then `capi.Gui.LoadOrUpdateCairoTexture` uploads it to a `LoadedTexture` (reusing the GL texture when size is unchanged) and `capi.Render.Render2DTexturePremultipliedAlpha` blits it to screen at the texture's actual integer pixel size. Cairo's `Format.Argb32` uses premultiplied alpha, matching the blit call.

**Bounds:** root = full dialog size (passed as `contentBounds` to `_builder.Render`). Each slot's bounds are computed by the layout pass: measure → position → recurse. See `05.layout-parameters.md` for full details.

**Dialog bootstrap:** `GuiDialog` creates `DialogRenderer`, attaches root `RenderHandle`. `DialogRenderer` creates and registers the `CairoDialogInputInterceptor` (minimal vanilla `GuiDialog` that keeps `DialogsOpened > 0` for cursor release and input routing) — the interceptor is owned entirely by the renderer and is disposed together with it. `Open()` calls `_renderer.TryOpen()`, fires initial `RequestReconcile()`.

**Focus & input routing:** the interceptor uses vanilla defaults — `ShouldReceiveMouseEvents` while opened, `ShouldReceiveKeyboardEvents` while focused. Vanilla `RequestFocus` calls `Focus()`/`UnFocus()` on the interceptor, which forwards to `DialogRenderer.OnFocus()`/`OnUnFocus()`, which updates the renderer's `_isFocused` flag and then calls `IGuiDialog.OnFocus()`/`OnUnFocus()` to propagate to `GuiDialog.IsFocused` and `OnFocusChanged`. The interceptor's event overrides call `DialogRenderer.OnMouse*()`/`OnKey*()` directly — the renderer is the first and only dispatch hub. Keyboard handlers first call `IGuiDialog.OnKey*()` for dialog-wide shortcuts, then forward unhandled events to `DialogInputDispatcher`. Mouse handlers forward to `DialogInputDispatcher`, which hit-tests recorded interactive regions and invokes the slot-level callbacks registered through builder extensions. Mouse events are only marked `Handled` when they hit the dialog or overlay interaction surface — clicks outside propagate to other dialogs / HUD / world. Mouse-down on a hit also calls `RequestFocus()`. Mouse wheel is consumed only when the dialog is focused AND hovered. Override `GuiDialog.OnFocusChanged(bool)` to react to focus changes.

**Edge resize:** `GuiDialog.IsResizable` (default `false`) enables drag-to-resize on the **bottom edge, right edge, and SE corner only** — top/left edges are deliberately excluded so the dialog's centre+offset positioning model doesn't need compensation. `MinWidth`/`MinHeight`/`MaxWidth`/`MaxHeight` (defaults 200/150/4096/4096) clamp the size; they have no effect while `IsResizable` is false. The grab band is 6 logical pixels wide along each edge; the SE corner is the square overlap. Three custom cursors are registered with the platform layer on first use (`GuiResizeCursors.EnsureLoaded`, called from the `IsResizable` setter): `bitzart-uitw-resize-h` (East), `bitzart-uitw-resize-v` (South), `bitzart-uitw-resize-nwse` (SE corner). The PNG assets ship at `resources/assets/bitzartuitweaks/textures/gui/cursors/` alongside a `coords.json` declaring hot-points — the same layout vanilla uses at `assets/game/textures/gui/cursors/` (vanilla itself ships only `normal`/`textselect`/`linkselect`/`move`/`busy`, no resize sprites). Loading mirrors `ScreenManager.LoadCursor` exactly: `IAssetManager.TryGet` → `IAsset.ToBitmap(api)` → `ClientPlatformAbstract.LoadMouseCursor` (reached via reflection in `ClientApiExtensions.LoadMouseCursor`). macOS is short-circuited like vanilla. Setting `MouseOverCursor` on the input interceptor each frame applies the cursor — vanilla `GuiManager.OnRenderFrameGUI` reads that field unconditionally, so during an active resize gesture the cursor stays correct even when the pointer moves outside the dialog rectangle. Mouse-down on a band starts a resize gesture handled directly in `GuiDialog.OnMouseDown` *before* slot dispatch; the gesture snapshots `Width`/`Height`/`OffsetX`/`OffsetY` at down-time and recomputes them from cursor delta on every move event. The screen-centred positioning means growing by `d` on the right pushes both edges out by `d/2`, so a counter-offset of `+d/2` on `OffsetX` keeps the left edge stationary; same for the bottom. Resize updates call `OnResizeUpdated(sizeChanged)`; the base implementation requests arrange when the fixed dialog size changed, while `ModConfigDialog` requests arrange on every update so arrange-time fractional child constraints are recalculated while dragging. The renderer's size/scale check still recreates the Cairo surface and requests the appropriate dirty state as a safety net.
