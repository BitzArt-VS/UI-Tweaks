# Keyboard Events & Focus

Keyboard input is focus-based, not spatially routed. Only the currently focused `IGuiNode` (tracked by `FocusManager`) receives keyboard events. Compare with mouse events, which hit-test against `InteractiveRegion` bounds each frame.

## Focus model

`FocusManager` is published at the dialog root as a cascading value. Any component that wants to receive keyboard input must request focus explicitly.

**Acquiring focus:** call `FocusManager.RequestFocus(this)` from an `OnMouseDown` handler (or any other event). The framework does not auto-focus on tab or any other heuristic.

**Losing focus:**
- `FocusManager.Blur()` — explicit programmatic blur.
- Mouse-down anywhere that does *not* call `RequestFocus` — the dispatcher auto-blurs after the full dispatch chain completes without a focus request.
- Dialog close.

**Checking focus state:** `FocusManager.IsFocused(node)` compares by reference identity. Components typically resolve `FocusManager` once in `OnParametersSet` and cache it, then call `IsFocused(this)` in `Render` to draw focus chrome.

**Caret blink:** the focused node receives `OnFrame(deltaTime)` each GUI frame. `GuiTextInput` uses that hook to flip its private caret blink flag on a fixed cadence (`BlinkPeriodSeconds = 0.5f`) and calls `RequestPaint()` when the caret visibility changes.

## API

**Two registration sites — both fire for the focused slot:**

1. **Own-slot configuration (`GuiNode.ConfigureSlot`)** — component-owned key callbacks registered through `IGuiSlotBuilder`; cleared and re-registered each pass:
   - `builder.OnKeyDown(Action<GuiKeyEventArgs>)` / `builder.OnKeyDown(Func<GuiKeyEventArgs, Task>)`
   - `builder.OnKeyUp(...)`
   - `builder.OnKeyPress(...)`

2. **Slot-level builder extensions (`KeyEventBuilderExtensions`)** — attach to `IGuiComponentBuilder<T>` during `BuildRenderTree`, cleared and re-registered each pass (same lifetime model as mouse handlers):
   - `.OnKeyDown<T>(Action<GuiKeyEventArgs>)` / `.OnKeyDown<T>(Func<GuiKeyEventArgs, Task>)`
   - `.OnKeyUp<T>(...)` (same two overloads)
   - `.OnKeyPress<T>(...)` (same two overloads)
   - `.OnFocusChanged<T>(Action<bool>)` / `.OnFocusChanged<T>(Func<bool, Task>)`

These registration sites are independent. Order: own-slot callback → external slot callback.

## `GuiKeyEventArgs`

Readonly struct wrapping vanilla `KeyEvent`. `Handled` writes propagate back to the inner `KeyEvent`, so multiple handlers in the same dispatch chain see the same flag.

| Property | Notes |
|---|---|
| `KeyCode` | GlKeys code of the pressed/released key |
| `KeyChar` | Printable character (meaningful only for `OnKeyPress`) |
| `ShiftPressed` / `CtrlPressed` / `AltPressed` / `CommandPressed` | Modifier state |
| `Handled` | Set to suppress framework fallbacks (see below) |
| `Event` | Raw vanilla `KeyEvent` — for advanced cases only |

## Dispatch pipeline

1. `CairoDialogInputInterceptor` receives keyboard events from vanilla's `GuiManager` and calls `DialogRenderer.OnKeyDown/Up/Press` directly.
2. `DialogRenderer` calls `IGuiDialog.OnKeyDown/Up/Press` for dialog-level virtual hooks (protected overrides on `GuiDialog` subclasses, e.g. Ctrl+S shortcuts). If the hook sets `args.Handled`, dispatch stops.
3. `DialogRenderer` delegates to `DialogInputDispatcher`: walks `_keyboardRegions` to find the region whose `Token` matches `FocusedNode`, then invokes its callbacks.

## Escape handling

`GuiDialog.OnEscapePressed()` (called by the interceptor before keyboard dispatch):
- If any node is focused → blur it and return `true` (consume without closing). This gives components one Escape press to cancel their active input before the dialog closes.
- If nothing is focused → close the dialog and return `true`.

Components that need to consume Escape themselves should mark `args.Handled = true` in their `builder.OnKeyDown` handler. The `GuiDialog` explicit implementation checks `args.Handled` before forwarding to the renderer, so a handled event from the dialog's own `OnKeyDown` override also suppresses the `DispatchKeyDown` call.

## Dialog-level overrides

`GuiDialog` exposes `protected virtual` overrides that fire *before* the dispatcher:

```csharp
protected virtual void OnKeyDown(KeyEvent args) { }
protected virtual void OnKeyUp(KeyEvent args) { }
protected virtual void OnKeyPress(KeyEvent args) { }
```

These are for dialog-wide shortcuts (e.g. Ctrl+S to save) that should fire regardless of which component is focused. Set `args.Handled = true` to prevent the event from reaching the focused component.
