# Mouse Events & Callbacks

Mouse handlers are attached to explicit render-tree slots:

- Declare callbacks during `BuildRenderTree` through render-tree-builder extensions.
- Component authors can register callbacks for the component's own allocated bounds from `ConfigureSlot(IGuiSlotBuilder)`.
- Wrap visible content in `GuiMouseTarget` when you need a layout-transparent hit region, then attach callbacks to that target.
- Layout-transparent `GuiNode` wrappers are declared with `Add<T>` just like components; their hit bounds are the union of their inlined children.

## API

**Slot-builder extensions (`MouseEventBuilderExtensions`):**

- `.OnMouseDown<T>(Action<GuiMouseEventArgs>)` / `.OnMouseDown<T>(Func<GuiMouseEventArgs, Task>)`
- `.OnMouseUp<T>(...)` (same two overloads)
- `.OnMouseClick<T>(...)` (same two overloads)
- `.OnMouseMove<T>(...)` (same two overloads)
- `.OnMouseEnter<T>(...)` (same two overloads)
- `.OnMouseLeave<T>(...)` (same two overloads)

Each event has separate sync (`Action<GuiMouseEventArgs>`) and async (`Func<GuiMouseEventArgs, Task>`) overloads — the framework dispatches synchronously regardless, but the async overload exists so callers can plug in `async Task` methods directly without wrapping. Internally both funnel through `GuiCallback<GuiMouseEventArgs>`. The extensions target `IGuiSlotBuilder`, so they work on external component slots and inside `ConfigureSlot`.

**Per-pass storage:** mouse handlers live on the slot's `RenderTreeFrame` and are cleared by `Reset()` at the start of each blueprint pass — same lifetime model as `Configure` actions. Re-register every pass via own-slot configuration or the extensions. Multiple callbacks for the same event are additive; own-slot callbacks run before external builder callbacks.

## `GuiCallback` / `GuiCallback<T>`

Value-type union of `Action`/`Func<Task>` (or `Action<T>`/`Func<T,Task>`). One `object?` reference + one `bool` flag — `default(GuiCallback)` means "no handler" and incurs zero allocation. Implicit conversions from both delegate flavours.

`Invoke()` dispatches synchronously; `InvokeAsync()` returns the task (or `Task.CompletedTask` for sync handlers / no handler).

**DX caveat:** lambdas like `() => Foo()` cannot bind to `GuiCallback` parameters via implicit conversion (compiler can't pick between `Action` and `Func<Task>` source). For public API entry points that take a click-style callback (e.g. `AddButton`), expose **two overloads** — one taking `Action`, one taking `Func<Task>` — both routing into a private core that takes a `GuiCallback`. This matches Blazor's overload pattern and keeps lambda use unambiguous.

## `GuiMouseEventArgs`

`GuiMouseEventArgs` is a `readonly record struct` with three primary fields:

- **`Position` (`GuiPoint`)** — dialog-local logical (unscaled) pixels; the same coordinate space `GuiComponentBounds` uses. Use this for all hit-tests and position-relative computations (scrollbar handle position, caret placement, etc.).
- **`AbsolutePosition` (`GuiPoint`)** — screen-absolute logical pixels (physical px / GUIScale from the screen's top-left corner). Use this when computing drag *deltas* that move the dialog — the dialog-local frame shifts with each position update but the screen-absolute frame stays stable.
- **`Button` (`EnumMouseButton`)** — which button triggered the event.
- **`WheelDelta` (`float`, init-only)** — non-zero only for wheel events.

`GuiPoint` is `readonly record struct GuiPoint(double X, double Y)` in `Public/Common/`.

**Rule of thumb:** use `Position` unless the handler *moves the dialog itself* (or moves something relative to the screen rather than the dialog surface), in which case use `AbsolutePosition`.

## Hit-test pipeline

1. **Region recording.** During the current combined arrange/paint walk in `GuiRenderTreeBuilder.Render`, after a slot's allocated bounds are resolved, the builder pushes an `InteractiveRegion` (bounds + handlers + slot instance as opaque token) into the `DialogInputDispatcher`'s shared region list when the slot has at least one registered mouse handler. Regions are recorded in render order; non-interactive slots are zero-cost.
2. **Hit-test.** `DialogInputDispatcher.HitTest(lx, ly)` walks the region list in **reverse** so later-rendered (z-order topmost) regions win.
3. **Coordinate conversion.** `DialogRenderer.TryToLogical(physX, physY, out lx, out ly)` converts physical-pixel screen coordinates to dialog-local logical pixels (the same space `GuiComponentBounds` uses).

## Mouse capture & semantics

- **`OnMouseDown`** fires when the cursor presses inside a region.
- **`OnMouseUp`** fires on the region that received the most recent `OnMouseDown` — *regardless of where the cursor is at release*. This is implicit mouse capture (Win32 SetCapture / browser pointer capture / vanilla VS button behaviour). Lets buttons reliably clear "pressed" visual state even when released outside.
- **`OnMouseClick`** fires only when the release lands inside the captured region's *current* bounds (looked up in the live region list, so layout changes between down and up are handled).
- **`OnMouseMove`** fires while a slot has mouse capture and on uncaptured hover movement while the cursor is inside the region.
- **`OnMouseEnter`** fires once when the uncaptured cursor first moves over the region.
- **`OnMouseLeave`** fires once when the uncaptured cursor leaves the region, or leaves the dialog after a previous enter.

The captured callbacks plus the slot identity (`InteractiveRegion.Token` = the slot's `IGuiNode` instance) are snapshotted on `MouseDown` and held in `DialogInputDispatcher` until `MouseUp`. The captured button gates re-entrancy across multiple buttons. Callbacks are cleared **before** invocation so a synchronous re-dispatch (rare but possible) doesn't see stale state.

## `GuiButton` worked example

`GuiButton` registers `OnMouseDown` / `OnMouseUp` / `OnMouseClick` for its own allocated bounds from `ConfigureSlot`. The button keeps a private `_isPressed` field updated by those handlers and reads it in `Render` for the pressed-overlay visual.

No `RequestReconcile` is needed for the press visual — `GuiButton` uses `RequestPaint`, so `_isPressed` is picked up by the next dirty paint without rebuilding the button subtree.

The button's outward API is just `Text`, `OnClick` (`GuiCallback`), and `Enabled` — no overridable mouse hooks. External callers configure clicks through:

- Inline: `b.AddButton(0, "Save", onClick: () => DoSave())` — the `Action` and `Func<Task>` overloads disambiguate lambdas.
- Fluent: `.OnClick(MyMethodGroup)` — also two overloads, friendly to method groups.

## Forwarding from `CairoDialogInputInterceptor`

`CairoDialogInputInterceptor.OnMouseDown(MouseEvent)` calls `_renderer.OnMouseDown(args)` directly. `DialogRenderer.OnMouseDown` applies the routing pipeline:

1. If the event is already handled, returns.
2. Dispatches to `DialogInputDispatcher.DispatchMouseDown`, which hit-tests recorded regions, captures the hit slot, and invokes its slot-level callback.
3. On a hit, requests focus and marks the vanilla event handled.

`OnMouseUp` always calls `DialogInputDispatcher.DispatchMouseUp` — even when the cursor is outside the dialog — so a captured component can release its pressed state on release-outside. Dialog title-bar dragging is a regular `GuiMouseTarget` in `GuiDialogTitleBar`; dialog edge resizing is registered by `DialogRenderer` as a root background region and forwarded through internal `IGuiDialog` mouse members.
